# EnerDEX — Full Project Report

**Date:** April 2026  
**Version:** 1.0 (Local Prototype)  
**Repository:** `BLC` (EnerDEX monorepo)

---

## 1. Executive Summary

EnerDEX is a full-stack decentralised energy trading platform built on Ethereum. It allows energy producers (sellers) and consumers (buyers) to trade tokenised energy units peer-to-peer, with on-chain escrow, automated fee collection, and a loyalty reward programme.

The system is composed of two coordinated sub-packages:

| Package | Role |
|---|---|
| `dem-project` | Hardhat + Solidity — smart contract layer |
| `dem-frontend` | React + Vite + Ethers v6 — user-facing dApp |

A unified deployment script links both packages: it compiles and deploys the contracts, then pushes ABIs and contract addresses directly into the frontend configuration, eliminating all manual synchronisation steps.

## 2. System Architecture

### 2.1 Blockchain Layer — `dem-project`

Built with **Hardhat 2.28.6** and **Solidity 0.8.28**, targeting a local `localhost` network at `http://127.0.0.1:8545`.

**Contracts:**

| Contract | Purpose |
|---|---|
| `EToken.sol` | ERC-20 energy token (`EnergyToken`, symbol `ETK`) with owner-controlled minting |
| `EnergyMarketplace.sol` | Escrow marketplace with fee logic, loyalty points, and tiered discounts |

**Key dependencies:**
- `@openzeppelin/contracts ^5.6.1` — `ERC20`, `Ownable`
- `@nomicfoundation/hardhat-toolbox ^6.1.2` — Ethers, Chai, coverage

---

### 2.2 Frontend Layer — `dem-frontend`

Built with **React 19**, **Vite 8**, and **Ethers v6**. Connects to MetaMask via `window.ethereum` using a `BrowserProvider`.

**Key source files:**

| Directory/File | Role |
|---|---|
| `src/context/WalletContext.jsx` | All contract interaction logic and state management |
| `src/components/`, `src/pages/` | Modular UI components and page routing |
| `src/App.css` | Complete styling layer (Dark & Light themes) |
| `src/contracts/config.js` | Reads `VITE_ETOKEN_ADDRESS` and `VITE_MARKETPLACE_ADDRESS` from `.env` |
| `src/contracts/EToken.json` | Token ABI — auto-generated by deploy script |
| `src/contracts/EnergyMarketplace.json` | Marketplace ABI — auto-generated by deploy script |

---

### 2.3 Integration Layer

The deployment script (`dem-project/scripts/deployMarketplace.js`) performs full end-to-end setup:

1. Deploys `EToken`
2. Deploys `EnergyMarketplace` (passing the token address)
3. Mints 500 ETK to `signers[1]` (the local test seller account)
4. Copies both compiled ABI files into `dem-frontend/src/contracts/`
5. Writes `dem-frontend/.env` with `VITE_ETOKEN_ADDRESS` and `VITE_MARKETPLACE_ADDRESS`

`dem-frontend/src/contracts/config.js` reads:

```js
export const ETOKEN_ADDRESS = import.meta.env.VITE_ETOKEN_ADDRESS;
export const MARKETPLACE_ADDRESS = import.meta.env.VITE_MARKETPLACE_ADDRESS;
```

This `.env` file is gitignored and must be regenerated on each fresh deployment.

## 3. Smart Contract Design

### 3.1 `EToken.sol`

Inherits from OpenZeppelin `ERC20` and `Ownable`.

- **Constructor:** mints `1000 × 10^18` tokens to the deployer
- **`mint(address to, uint256 amount)`:** owner-only; mints `amount × 10^18` tokens (callers pass human units, e.g. `500`)

> **Important:** The `amount` parameter is in human-readable token units, not wei-scaled base units. The contract applies the `10^18` multiplier internally.

---

### 3.2 `EnergyMarketplace.sol`

Inherits from OpenZeppelin `Ownable`.

**State:**

| Variable | Type | Description |
|---|---|---|
| `energyToken` | `EToken` | Reference to deployed EToken contract |
| `listings` | `mapping(uint256 => Listing)` | All listings by ID |
| `listingCount` | `uint256` | Total listings ever created |
| `loyaltyPoints` | `mapping(address => uint256)` | Points earned per address |
| `collectedFees` | `uint256` | Accumulated platform fee balance (in wei) |
| `FEE_PERCENT` | `uint256` constant | 2 |
| `POINTS_PER_TRADE` | `uint256` constant | 10 |

**Events:**

| Event | Emitted When |
|---|---|
| `EnergyListed(listingId, seller, amountETK, priceWei)` | A new listing is created |
| `EnergyTraded(listingId, buyer, seller, amountETK, priceWei)` | A purchase completes |
| `ListingCancelled(listingId)` | A listing is cancelled |

**Functions:**

| Function | Access | Description |
|---|---|---|
| `listEnergy(amountETK, priceWei)` | External | Escrows seller's ETK and creates a listing |
| `buyEnergy(listingId)` | External payable | Purchases a listing; applies discount; transfers ETK and ETH |
| `cancelListing(listingId)` | External | Seller cancels their own listing; returns escrowed ETK |
| `withdrawFees()` | Owner-only | Transfers all collected platform fees to the owner |
| `getDiscount(address)` | Public view | Returns the buyer's current fee discount percentage |
| `getListing(listingId)` | External view | Returns the full details of a listing |

**Fee and discount calculation (in `buyEnergy`):**

```
discount = getDiscount(buyer)           // 0, 10, 25, or 50
fee      = (priceWei × 2 × (100 − discount)) / 10000
seller   = priceWei − fee
```

State is marked inactive **before** ETK/ETH transfers, following the checks-effects-interactions security pattern.

**Loyalty discount tiers:**

| Loyalty Points | Fee Discount |
|---|---|
| < 10 | 0% |
| ≥ 10 | 10% |
| ≥ 50 | 25% |
| ≥ 100 | 50% |

## 4. Business Flow

### 4.1 Listing

1. Seller enters ETK amount and ETH price in the frontend.
2. Frontend calls `EToken.approve(marketplace, amountETK × 10^18)`.
3. Frontend calls `EnergyMarketplace.listEnergy(amountETK, priceWei)`.
4. Contract pulls `amountETK × 10^18` token units into escrow and stores the listing.

### 4.2 Purchase

1. Buyer selects an active listing in the frontend.
2. Frontend calls `EnergyMarketplace.buyEnergy(listingId)` with exact ETH value.
3. Contract validates listing state, buyer identity, and ETH amount.
4. Discount is applied to compute the platform fee.
5. Listing is marked inactive; fees are accumulated.
6. `amountETK × 10^18` tokens are transferred to the buyer.
7. ETH minus fee is transferred to the seller.
8. Both buyer and seller receive 10 loyalty points.

### 4.3 Cancellation

1. Seller calls `cancelListing(listingId)`.
2. Contract validates that the caller is the listing owner and the listing is active.
3. Listing is marked inactive; escrowed ETK is returned to the seller.

### 4.4 History and Analytics

1. Frontend queries `EnergyListed` and `EnergyTraded` event logs alongside block timestamps.
2. Events are merged and sorted by timestamp for human-readable dates.
3. Seller income summary is computed from `EnergyTraded` events.

## 5. Test Coverage

All tests are in `dem-project/test/` and run with `npm test` (Hardhat + Chai).

### `EToken.test.js`

- Deployer receives initial 1000 ETK supply
- Token name is `EnergyToken`, symbol is `ETK`
- Owner can mint tokens successfully
- Non-owner mint call reverts

### `EnergyMarketplace.test.js`

- Listing creation sets correct state and active flag
- Buy flow correctly transfers ETK to buyer and ETH to seller
- 2% platform fee is deducted and accumulated in `collectedFees`
- Loyalty points are accrued for buyer and seller after each trade
- Discount is applied correctly when buyer has sufficient points
- Listing cancellation returns escrowed ETK to seller
- Revert on attempt to buy an inactive listing
- Revert on seller attempting to buy their own listing
- Revert on incorrect ETH amount sent
- Owner fee withdrawal clears `collectedFees` and transfers ETH

## 8. Quality Assessment

### Strengths

- Complete end-to-end architecture with clear separation between contract and frontend layers
- Practical marketplace mechanics: escrow, fee collection, loyalty rewards, tiered discounts
- Comprehensive Hardhat test suite covering all critical business paths and revert conditions
- Automated deployment pipeline that eliminates manual ABI/address synchronisation
- Security-conscious contract design: state updated before external calls (checks-effects-interactions)
- Clean Hardhat configuration targeting explicit solidity version and localhost network

### Known Limitations and Improvement Areas

| Area | Current State | Suggested Improvement |
|---|---|---|
| Frontend testing | No automated test suite | Add Vitest + React Testing Library |
| Event indexing | Queries from block `0` | Add start-block optimization or use an indexer (e.g. The Graph) |
| Reentrancy protection | `ReentrancyGuard` implemented | Strong security pattern applied |
| Marketplace flexibility | Fixed 2% fee and fixed point values | Make fee and points configurable by the owner |
| Error handling | Basic `require` statements | Add custom Solidity errors for gas efficiency |

---

## 9. Conclusion

EnerDEX is a functional, end-to-end prototype of a decentralised energy marketplace. It demonstrates core Web3 mechanics — token issuance, escrow trading, on-chain fee logic, and event-driven analytics — within a reproducible local development environment. The automated deployment pipeline makes the system straightforward to set up and iterate on. The natural next step is refactoring the frontend into modular components and adding frontend automated testing before any public deployment.
