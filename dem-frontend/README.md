# dem-frontend

> React + Vite dApp frontend for EnerDEX.

This package provides the user-facing interface for the EnerDEX platform. It connects to the `EToken` and `EnergyMarketplace` contracts via MetaMask and exposes full trading, analytics, and admin functionality.

---

## Features

- Wallet connection and account management (MetaMask)
- Real-time ETH and ETK balance display
- Create, browse, and cancel energy listings
- Purchase listings with exact ETH payment
- Loyalty points and fee discount tier display
- Event-driven transaction history view
- Seller income analytics from on-chain events
- Owner admin panel: mint ETK, withdraw collected fees

## Contract Address Configuration

`src/contracts/config.js` reads deployed addresses from Vite environment variables:

```js
export const ETOKEN_ADDRESS      = import.meta.env.VITE_ETOKEN_ADDRESS;
export const MARKETPLACE_ADDRESS = import.meta.env.VITE_MARKETPLACE_ADDRESS;
```

These values are written automatically to `dem-frontend/.env` by `dem-project/scripts/deployMarketplace.js` when you run `npm run deploy`. You do not need to set them manually.

> **Note:** `.env` is gitignored. It must be regenerated each time contracts are redeployed.

## Setup and Local Workflow

### Prerequisites

- MetaMask browser extension installed
- `dem-project` contracts deployed and running on a local Hardhat node

### Step 1 ‚Äî Start the local blockchain and deploy contracts

```bash
# Terminal A ‚Äî start node
cd ../dem-project
npm run node

# Terminal B ‚Äî deploy and auto-sync this package
cd ../dem-project
npm run deploy
```

### Step 2 ‚Äî Install dependencies and start the dev server

```bash
cd dem-frontend
npm install
npm run dev
```

Open the Vite dev URL (default: `http://localhost:5173`), connect MetaMask to **Localhost 8545**, and import a test account using a private key from the Hardhat node output.

## Contract Interaction Summary

### Read operations (`loadData`)

| Data | Source |
|---|---|
| ETH balance | `provider.getBalance(account)` |
| ETK balance | `EToken.balanceOf(account)` |
| Loyalty points | `EnergyMarketplace.loyaltyPoints(account)` |
| Fee discount | `EnergyMarketplace.getDiscount(account)` |
| Owner status | `EToken.owner()` == account |
| Collected fees | `EnergyMarketplace.collectedFees()` |
| Active listings | `getListing(i)` for `i` in `[0, listingCount)` |

### Event queries (`loadHistory`)

| Event | Source |
|---|---|
| `EnergyListed` | `EnergyMarketplace` event log |
| `EnergyTraded` | `EnergyMarketplace` event log |

Events are merged and sorted by block number. Seller income is computed from `EnergyTraded` events for the connected account.

### Write operations

| Function | Action |
|---|---|
| `listEnergy()` | Calls `EToken.approve(marketplace, amount)` then `listEnergy(amountETK, priceWei)` |
| `buyEnergy(listingId)` | Calls `buyEnergy(listingId)` with exact ETH value |
| `cancelListing(listingId)` | Calls `cancelListing(listingId)` |
| `withdrawFees()` | Owner-only; calls `withdrawFees()` |
| `mintTokens(to, amount)` | Owner-only; calls `EToken.mint(to, amount)` |

## MetaMask / Ethers Integration

- Provider: `BrowserProvider(window.ethereum)` ‚Äî read-only calls
- Signer: `provider.getSigner()` ‚Äî used for all write transactions
- Contract construction is centralised in a `getContracts()` helper used by all write actions

---

## Unit and Conversion Conventions

| Operation | Convention |
|---|---|
| Display ETK balance | `formatUnits(rawValue, 18)` |
| Approve ETK for listing | `parseUnits(sellAmount, 18)` |
| `listEnergy` call | Passes human ETK units (e.g. `50`) ‚Äî marketplace scales internally |
| `mint` call | Passes human units (e.g. `500`) ‚Äî token contract scales internally |

## UI Behaviour

- A **welcome / connect screen** is shown when no wallet is connected
- Tabbed interface: **Dashboard**, **Market**, **History**, **Income**, **Loyalty**
- Transaction status notifications track the full lifecycle: pending ‚Üí confirmed / failed
- **Owner admin panel** is shown only when the connected account matches the on-chain `owner()`
- Address and transaction hash copy helpers are available throughout the UI

---

## Notes

- History queries start from block `0`. This is appropriate for local and demo deployments; it requires optimisation (e.g. block-range filters) for production chains.
- ABIs in `src/contracts/` are auto-generated ‚Äî do not edit them manually.
- Re-running `npm run deploy` regenerates `.env` and the ABI files; restart the dev server afterwards to pick up the new addresses.
