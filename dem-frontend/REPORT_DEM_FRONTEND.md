# dem-frontend — Detailed Technical Report

**Package:** `dem-frontend`  
**Role:** React dApp user interface for EnerDEX  
**Date:** April 2026

---

## 1. Scope

`dem-frontend` is the user-facing decentralised application (dApp) for EnerDEX. It provides a browser-based interface for interacting with the `EToken` and `EnergyMarketplace` smart contracts via MetaMask.

Responsibilities:
- Wallet authentication and account management
- Displaying real-time on-chain balances, listing data, loyalty status, and fee information
- Executing contract write transactions (list, buy, cancel, mint, withdraw)
- Presenting transaction history and seller income analytics derived from contract events

## 4. Environment and Address Configuration

`src/contracts/config.js` resolves contract addresses at runtime from Vite environment variables:

```js
export const ETOKEN_ADDRESS      = import.meta.env.VITE_ETOKEN_ADDRESS;
export const MARKETPLACE_ADDRESS = import.meta.env.VITE_MARKETPLACE_ADDRESS;
```

`dem-frontend/.env` is written automatically by `dem-project/scripts/deployMarketplace.js`:

```
VITE_ETOKEN_ADDRESS=0x...
VITE_MARKETPLACE_ADDRESS=0x...
```

This replaced an older pattern that required manually editing a static address file. The `.env` is gitignored; it must be regenerated on every fresh contract deployment.

## 6. Contract Access Layer

### 6.1 Read — `loadData()`

Called on wallet connection and after each write transaction.

| Data Fetched | Contract Call |
|---|---|
| ETH balance | `provider.getBalance(account)` |
| ETK balance | `etoken.balanceOf(account)` |
| Loyalty points | `marketplace.loyaltyPoints(account)` |
| Fee discount | `marketplace.getDiscount(account)` |
| Is owner | `etoken.owner() === account` |
| Collected fees | `marketplace.collectedFees()` |
| Listings | `marketplace.listingCount()` → loop `getListing(i)` for active entries |

### 6.2 Read — `loadHistory()`

Queries event logs for the full transaction history.

| Event | Source |
|---|---|
| `EnergyListed` | Marketplace contract event log, all blocks from `0` |
| `EnergyTraded` | Marketplace contract event log, all blocks from `0` |

Events are merged, sorted by block number, and rendered in the History tab. Seller income is computed from `EnergyTraded` events where the seller matches the connected account.

### 6.3 Write — `getContracts()`

A centralised helper that constructs signer-connected contract instances:

```js
const provider = new BrowserProvider(window.ethereum);
const signer   = await provider.getSigner();
const etoken   = new Contract(ETOKEN_ADDRESS, ETokenABI, signer);
const market   = new Contract(MARKETPLACE_ADDRESS, MarketplaceABI, signer);
return { etoken, market };
```

Used by all write actions to avoid repeated provider/signer construction.

### 6.4 Write Actions

| Function | Steps |
|---|---|
| `listEnergy()` | `etoken.approve(marketplace, parseUnits(amount, 18))` → `market.listEnergy(amountETK, priceWei)` |
| `buyEnergy(id)` | `market.buyEnergy(id, { value: listing.priceWei })` |
| `cancelListing(id)` | `market.cancelListing(id)` |
| `withdrawFees()` | `market.withdrawFees()` |
| `mintTokens()` | `etoken.mint(mintAddress, mintAmount)` (passes human units) |

## 7. Unit and Conversion Conventions

| Operation | Convention Used |
|---|---|
| Display ETK balance | `formatUnits(rawBalance, 18)` |
| `approve` for listing | `parseUnits(sellAmount, 18)` (converts to base units for the token contract) |
| `listEnergy` call | Human token units (e.g. `50`) — marketplace contract handles base unit conversion |
| `mintTokens` call | Human token units (e.g. `500`) — EToken contract handles base unit conversion |

> **Important:** `EToken.mint` multiplies by `10^18` internally. The `parseUnits` conversion is only used for the ERC-20 `approve` call; it must **not** be used when calling `listEnergy` or `mint` directly.

## 8. UX and Interface Design

- **Disconnected state:** Welcome screen with a prominent connect-wallet call to action
- **Tabbed navigation:** Dashboard · Market · History · Income · Loyalty
- **Status notifications:** Real-time messages track each transaction from submission through confirmation or failure
- **Owner admin panel:** Rendered only when `isOwner === true` (verified via `etoken.owner()` on-chain)
- **Convenience helpers:** One-click address and transaction hash copy across all views

## 9. Reproducible Local Setup

```bash
# Step 1 — Start blockchain node
cd dem-project && npm run node

# Step 2 — Deploy contracts and sync frontend
cd dem-project && npm run deploy

# Step 3 — Start frontend
cd dem-frontend && npm install && npm run dev
```

Connect MetaMask to **Localhost 8545** and import a Hardhat test account. The second account (`signers[1]`) is pre-funded with 500 ETK by the deploy script, making it ready to use as a test seller immediately.

## 10. Strengths

- Full feature coverage for the EnerDEX MVP use cases
- Clear transaction feedback during all write operations
- Owner/admin role gated correctly on-chain (not merely by UI flags)
- Event-driven history and income tracking with no off-chain database
- Automated ABI and address sync eliminates the most common source of integration errors
- Clean separation between read (`loadData`, `loadHistory`) and write (`getContracts` + action functions)

## 11. Known Limitations and Recommendations

| Item | Current State | Recommendation |
|---|---|---|
| Code organisation | All logic in a single `App.jsx` | Refactor into dedicated components (Market, History, Dashboard) and custom hooks (`useContracts`, `useWallet`) |
| Frontend testing | No automated test suite | Add Vitest + React Testing Library |
| Event query range | Queries from block `0` | Parameterise with a deployment block to reduce query time as chain grows |
| Error handling | Generic catch blocks | Improve user-facing error messages; distinguish contract reverts from network errors |
| Wallet assumptions | Only MetaMask (`window.ethereum`) | Support EIP-1193 compatible wallets via a library like wagmi or Web3Modal |
| `.env` lifecycle | Must be manually regenerated after each deploy | Document clearly; consider a post-deploy notification or check |

---

## 12. Summary

`dem-frontend` is a functional, production-style dApp interface for the EnerDEX smart contracts. It covers the full user journey from wallet connection through listing, trading, cancellation, and analytics. The architecture is well-suited to its current scope; the natural evolution path is to decompose `App.jsx` into smaller components and hooks, add an automated test suite, and integrate a broader wallet provider library for production deployments.
